This article is number 2 in a series where I establish a non-prod cluster at home.

Last time - we formulated a strategy for building a non-prod cluster, the first step being focused on infrastructure. In today's blog post, we'll implement that first step and I'll show my work along the way.

At the end of this article, you'll be able to see how I was able to set up bare-metal kubernetes cluster with `terraform`, `ansible`, and `proxmox`, and the series of steps I took along the way.

Here are the steps I took as well as my thought process.

## Step 1 - Experiment with Terraform & Proxmox

Ok. So, we're not starting from completely from scratch. I do have some code written in ansible to configure VMs, but I have no code established in version control to bring them into existence. Many of the VMs are created from a template, and to make matters worse, the template VM that I'm using is also just a series of steps in my notebook (ie. not scripted out yet).

I'm thinking this template VM needs to be managed in a codebase in some way. I haven't worked much with `terraform` before, but I know I'm interested to try it out. Maybe managing this template VM with terraform could be a reasonable thing to try?

Well, after a few hours of trying to convert proxmox code in my journal to terraform, I came to a realization.

## Step 2 - Discover two vm management strategies

### Strategy 1 - Go full cloud

One option is that I create the template VM in proxmox. Then, I create lots of VMs from that template. Once VMs are created from the template, I then configure all of them using ansible.

On one hand, this option is attractive, as its highly scriptable. Everything can be run in sequence via scripts.

On the other hand, I have to now run tons of configuration code against tons of VMs, which opens up opportunity for things to go wrong. When things go wrong, do I delete the whole fleet and start again or SSH onto a buggy VM and turn it into a snowflake. This approach requires very high confidence in the VM configuration code (ansible) which I presently don't have.

### Strategy 2 - Make a golden image

The other option is to bake configuration into the template. At one of my previous workplaces we called this making a vm "snapshot."

This approach, I do like. If I can make a template VM in proxmox, I can start it up, and configure it - mounting storage, installing packages, and getting it nicely prepared. With a "snapshot" or a "golden image" I can use it to make lots of VMs and not worry about making a mistake across them.

Moreover, when it finally comes time to upgrade to a new version of Ubuntu, I can work on a new template and roll it out in as a newly versioned "thing." As opposed to changing existing VMs across the board.

## Step 3 - Rework my approach to template management

Ok, So I've decided I want to go with the "golden image" approach.

Experimenting with terraform was great! I learned I can set all sort of things in proxmox, but I realized I'm not going to want to have terraform manage my vm templates. That is, if I run `terraform apply` I want my fleet of VMs to be created. Though, if I bundle the template into terraform, it would also be created in `terraform apply` and I wouldn't have any time to configure it before its used to create the fleet!

With all of this in mind, I made a choice to manage my template VM entirely in ansible. I wrote four playbooks to create a VM, start it, configure it, and then cleanly shut it down, finalizing it for replication. If you're interested, you can check out these playbooks here:

home-playbooks / playbooks / [ansible-template-ubuntu-noble](https://github.com/pbabbott/home-playbooks/tree/main/playbooks/ansible-template-ubuntu-noble)

So the strategy is like this:

- 1 - Ansible makes VM on Proxmox host
- 2 - Ansible Starts the VM (target: Proxmox)
- 3 - Ansible connects to newly created VM and configures it, with a clean shutdown
- 4 - Ansible converts VM to be a template (target: Proxmox)
- 5 - `terraform apply` uses golden image to make fleet (target: Proxmox)
- 6 - Ansible further configures fleet

### Template Configuration vs Fleet Configuration

Now you might be wondering, in step 3 we configure a vm and in step 6 we configure many vms. Why configure the VMs twice?

My thinking is as follows:

In the template VM (golden image), we configure things like storage mounts, iptables, dns, ipv6, timezone, etc. All of these things are very unlikely to change. I'm not go change timezones anytime soon, as these servers are in my house. Storage mounts would only change with new hardware, and my hardware has no more slots for more drives.

In the fleet VMs, I'm thinking about apps. I know that `kubernetes` and `containerd` both release new version upgrades on a cadence. Tools like `kubectl` and `helm` also change frequently.

Ultimately either place is valid for managing configuration. Its just about anticipating what will need to be upgraded.

## Step 4 - Speed Bump: I rebuilt my home network

Ok. So while I was working on all of this, I found myself deleting VMs and re-creating them over-and-over again. Each time I would make a VM or many VMs, it would require me to go into my router and find their IP addresses so that I could target them with scripts. That is, each VM was set up with cloud init to use `ip=dhcp`.

### Problem: My IPs keep changing!

While this did work, it was also super annoying. I'd have to go into my Eero router app, and find the VM, and then give it an IP reservation so that the IP would not get changed. Even so, re-creating the VM gives it a new MAC and therefore a new IP. I needed a solution that would let me develop against a static IP.

### Start tinkering

To address this problem, I started messing about with the advanced networking settings in my Eero router app. I went to:

Settings > Advanced Networking > DHCP & NAT.

Then I changed it from `Automatic` (which would mean all DHCP all the time) to `Manual IP`. This lets you edit your Eero network IP range and DHCP range.

So, funny story, I started changing things in here without full understanding of the implications of each of the fields. If anything goes wrong, I assumed I could just set it back to `Automatic` - YOLO!

Unfortunately, my modem was not in pass-through (or bridge) mode. This caused my network to have a double-NAT and everything was under `192.168.X.X`. This ultimately caused my internet to go down, and I had a heck-of-a-time to getting it fixed. I did a hard-reset on my router and then I called Eero support and they were super helpful in talking through each of the settings with me. They stayed on the call with me until my internet came back up and also helped me solve my double-NAT issue. Honestly a great customer support experience!

Thankfully I got the internet back up before the kids wanted to watch their morning TV shows.

### Solution: DHCP + Static

Here's what I landed on.

My Eero network IP starts at `192.168.4.0`. And it has a subnet mask of `255.255.252.0`.

This means I can have IPs in the range of: `192.168.[4-7].[0-255]`.

This gives me 1024 IP addresses, which is more than enough for my house!

I also setup my router to only use DHCP in this range: `192.168.4.20` to `192.168.5.255`. The range `192.168.4.[0-19]` will stay reserved for various nodes of my Eero mesh network.

The `.6` and `.7` range will be used for static IPs.

### 5. A - Create a VM in proxmox using a shell script

While there were several ansible playbooks (create, start, configure, finalize), I want so share the the shell script I used to create the initial VM in proxmox.

This shell script is copied to my proxmox host using ansible and then used to create the VM.

```sh
#!/usr/bin/env bash
set -euo pipefail

# ------------------------------------------------------------
# Ubuntu Cloud-Init Template Bootstrap for Proxmox
# ------------------------------------------------------------
# This script creates a reusable Ubuntu cloud-init template.
# Terraform will clone from this template.
# Run once per Ubuntu release.
# ------------------------------------------------------------

TEMPLATE_ID="${TEMPLATE_ID:-901}"
TEMPLATE_NAME="${TEMPLATE_NAME:-ansible-template-ubuntu-noble}"
STORAGE="local-lvm"
BRIDGE="vmbr0"
MEMORY=6144
BALLOON=512
CORES=4
DISK_SIZE="32G"

CIUSER="${CIUSER:-admin}"
CIPASSWORD="${CIPASSWORD:-change-me}"

UBUNTU_IMAGE_URL="https://cloud-images.ubuntu.com/noble/current/noble-server-cloudimg-amd64.img"
IMAGE_FILE="noble-server-cloudimg-amd64.img"

### PRECHECKS ###
if qm status "$TEMPLATE_ID" &>/dev/null; then
  echo "VMID $TEMPLATE_ID already exists. Aborting."
  exit 1
fi

### DOWNLOAD IMAGE ###
if [ ! -f "$IMAGE_FILE" ]; then
  echo "Downloading Ubuntu cloud image..."
  wget -O "$IMAGE_FILE" "$UBUNTU_IMAGE_URL"
else
  echo "Image already downloaded."
fi

echo "Creating VM..."
qm create "$TEMPLATE_ID" \
  --name "$TEMPLATE_NAME" \
  --memory "$MEMORY" \
  --cores "$CORES" \
  --net0 virtio,bridge="$BRIDGE" \
  --scsihw virtio-scsi-pci \
  --ostype l26

echo "Setting memory balloon to $BALLOON..."
qm set "$TEMPLATE_ID" --balloon "$BALLOON"

echo "Importing disk..."
qm importdisk "$TEMPLATE_ID" "$IMAGE_FILE" "$STORAGE"

echo "Attaching disk..."
qm set "$TEMPLATE_ID" \
  --scsi0 "$STORAGE:vm-${TEMPLATE_ID}-disk-0,iothread=1,ssd=1" \
  --boot order=scsi0 \
  --bootdisk scsi0

echo "Resizing disk to $DISK_SIZE..."
qm resize "$TEMPLATE_ID" scsi0 "$DISK_SIZE"

echo "Adding cloud-init drive..."
qm set "$TEMPLATE_ID" --ide2 "$STORAGE:cloudinit"

echo "Disabling QEMU agent..."
qm set "$TEMPLATE_ID" --agent enabled=0

echo "Setting serial console..."
qm set "$TEMPLATE_ID" --serial0 socket --vga serial0

echo "Enabling auto-start..."
qm set "$TEMPLATE_ID" --onboot 1

echo "Setting user and password..."
qm set "$TEMPLATE_ID" --ciuser "${CIUSER}" --cipassword "${CIPASSWORD}"

echo "Setting SSH public key..."
qm set "$TEMPLATE_ID" --sshkey "/root/id_ed25519.pub"

echo "Setting static IP config..."
qm set "$TEMPLATE_ID" --ipconfig0 "ip=192.168.6.91/22,gw=192.168.4.1"

echo ""
echo "Template $TEMPLATE_NAME (VMID $TEMPLATE_ID) created successfully."
echo "Note: template is not yet finalized."
```

### 5. B - Configure the VM

Next, I want to show a high-level `main.yml` playbook that configures the VM. This one is responsible for turning it into that "golden image" or creating the "snapshot." The last step in this one shuts it down cleanly so that it is ready to be replicated across the fleet.

```yaml
---
# Configure the VM (template) as per playbooks/proxmox/README.md "Configure the VM".
# Targets the running template VM. Run after the template VM is started and
# reachable (e.g. at 192.168.6.91).

- name: Include QEMU guest agent play
  ansible.builtin.import_playbook: 01-qemu-guest-agent.yml

- name: Include memory ballooning play
  ansible.builtin.import_playbook: 02-memory-ballooning.yml

- name: Include Longhorn data disk play
  ansible.builtin.import_playbook: 03-longhorn-data-disk.yml

- name: Include DNS play
  ansible.builtin.import_playbook: 04-dns.yml

- name: Include timezone play
  ansible.builtin.import_playbook: 05-timezone.yml

  ansible.builtin.import_playbook: 06-iptables-ipv6.yml

- name: Include storage packages play (NFS, iSCSI)
  ansible.builtin.import_playbook: 07-storage-packages.yml

- name: Include clean shutdown and power off play
  ansible.builtin.import_playbook: 08-clean-shutdown.yml
```

## Step 6 - Create the fleet using terraform

Finally, I got terraform set up to create a fleet of VMs from the template.

I experimented with a few different providers for proxmox, but the one that I found to be the most stable (and least buggy) was the [BPG Proxmox provider](https://registry.terraform.io/providers/bpg/proxmox/latest/docs)

If you want to see my whole terraform solution, check it out here:

home-playbooks / [terraform](https://github.com/pbabbott/home-playbooks/tree/main/terraform)

The most exciting part of the code is that I managed to get it down to just a few variables to control instances.

```terraform
variable "nonprod_vms" {
  description = "Non-prod VMs: map of name -> vmid (301 = controller, 302-304 = workers). Use tf-nonprod-* names."
  type        = map(number)
  default = {
    "tf-nonprod-k8s-controller-1" = 301
    "tf-nonprod-k8s-worker-1"     = 302
    "tf-nonprod-k8s-worker-2"     = 303
    "tf-nonprod-k8s-worker-3"     = 304
  }
}

variable "nonprod_static_ips" {
  description = "Static IP address per non-prod VM (keys must match nonprod_vms). Used with /22 and nonprod_gateway."
  type        = map(string)
  default = {
    "tf-nonprod-k8s-controller-1" = "192.168.6.31"
    "tf-nonprod-k8s-worker-1"     = "192.168.6.32"
    "tf-nonprod-k8s-worker-2"     = "192.168.6.33"
    "tf-nonprod-k8s-worker-3"     = "192.168.6.34"
  }
}
```

And after running `terraform apply` they of course showed up in my proxmox host!

I now have many VMs constructed from my golden image. Now the last thing to do is to start installing cool stuff onto them. In particular, I'm going to use these to run `containerd` and a `kubelet` via `kubeadm`.

Below is the `main.yml` playbook i used to run many playbooks against the fleet.

Since I already had a suite of playbooks to configure my existing cluster, I decided to put all my of my new code into a `gen2` folder. That way, I wouldn't risk messing anything up with my first approach.

If you want to check it out for yourself its all here:

home-playbooks / playbooks / [kuberentes-gen2](https://github.com/pbabbott/home-playbooks/tree/main/playbooks/kubernetes-gen2)

## Reflection

Phew! What a journey! We did it!

I configured a new pattern for dynamic and static IP addresses at home. I've refactored my [home-playbooks](https://github.com/pbabbott/home-playbooks) repository. I got a new fleet of VMs established with proxmox. My notes are now runnable scripts. And we got it all done without messing with the existing setup.

While these accomplishments feel great, what feels even better is the learning I gained from the mistakes I made. I shouldn't have messed around with my home network as I don't have a non-prod eero router. I could have knocked out my home internet for quite a while. And honestly, this is exactly why i'm building a non-prod clsuter. I love to tinker and mess around with things even if they break. I want my production cluster to feel like production and not go down when i can't help but try new things.

I also accidentially discovered multiple strategies for handling VMs. Golden images or a cloud-based approach. I could see the benefit of having everything fully scripted and perhaps automatiable out with a full cloud approach, but my homelab codebase just isn't there yet. I think the approach I used for my homelab works well for my needs right now. I can destroy everything and re-create with a handful of manual steps. Things that change are now easier to change. I accomplished the goal of my first milestone. Ensure things can be re-run safely, and build confidence in my infrastructure management code.

Perhaps most exciting was that final moment of running `kubectl get nodes`. I am now fully ready to start Milestone 2. Next up, it'll be time to re-work the ingress pattern with `haproxy` and a transition from `ingress-nginx` to `istio`.
